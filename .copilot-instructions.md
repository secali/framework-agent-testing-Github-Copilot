# 🐱 GitHub Copilot Instructions for CAT (Copilot Agent Testing)

Este archivo proporciona instrucciones de contexto, arquitectura y reglas de diseño para que cualquier asistente de IA (GitHub Copilot, Cursor, etc.) comprenda y desarrolle de forma óptima en este repositorio.

---

## 📖 Contexto del Proyecto

**CAT (Copilot Agent Testing)** es un framework BDD (Behavior Driven Development) low-code diseñado para automatizar y validar el comportamiento de agentes de Inteligencia Artificial (especialmente GitHub Copilot Extensions y asistentes autónomos) sin necesidad de escribir código recurrente.

El framework parsea archivos `.feature` de Gherkin y ejecuta un conjunto de 47 steps pre-construidos que interactúan con el agente a través de diferentes adaptadores (`http`, `cli`, `mock`).

## ⚙️ Reglas de Codificación Clave (ESM Obligatorio)

> [!IMPORTANT]
> El proyecto está configurado en modo **ES Modules (ESM)** nativo en Node.js (`"type": "module"` en `package.json`).

1. **Importaciones Locales:** Siempre debes incluir la extensión `.js` en las rutas relativas al importar módulos TypeScript de este proyecto.
   * **Correcto:** `import { Given } from "./step-registry.js";`
   * **Incorrecto:** `import { Given } from "./step-registry";`
2. **Importaciones de Terceros:** Importa los módulos de npm de forma estándar (ej. `import Ajv from "ajv";`).

---

## 🧩 Reglas de Diseño de Pruebas (Gherkin BDD)

1. **Estrategia de Fixtures Externos (Clean Features):**
   * Evita incluir bloques grandes de texto, código de ejemplo, o JSON schemas directamente dentro de los archivos `.feature` usando `"""` (DocStrings).
   * **Usa siempre la estrategia de archivos externos** apuntando a la carpeta `fixtures/` mediante los steps dedicados:
     * `Given the system prompt from file "path"`
     * `Given the mock responds to "pattern" with file "path"`
     * `When I send the prompt from file "path"`
     * `Then the response matches the JSON schema in file "path"`
     * `Then the response matches the content of file "path"`

2. **Expresiones Regulares (Patrones de Texto):**
   * El validador regex implementado en `built-in-steps.ts` soporta flags estándar de JavaScript mediante la notación `/patrón/flags`.
   * Para aserciones case-insensitive en informes largos, prefiere siempre la sintaxis de barra inclinada con el flag `i`:
     * **Correcto:** `Then the response matches the pattern "/(causa raíz|error)/i"`
     * **Evitar:** `Then the response matches the pattern "(?i)(causa raíz|error)"` (no soportado nativamente en el constructor básico de JS).

## 🏗️ APIs y Clases Principales para Extensión

### 1. Registro de Steps Custom
Los steps se declaran importando `Given`, `When` o `Then` desde `src/gherkin/step-registry.ts`.
El tipo de datos del contexto de prueba que se pasa al callback es `TestContext` (definido en `src/types.ts`).

```typescript
import { Then } from "../src/gherkin/step-registry.js";
import type { TestContext } from "../types.js";

Then(
  "the response is in capital letters",
  (ctx: TestContext) => {
    const text = ctx.response?.text || "";
    const passed = text === text.toUpperCase();
    ctx.results.push({
      evaluator: "uppercase",
      label: "response is uppercase",
      passed,
      message: passed ? "✓ Text is in uppercase" : "✗ Text is NOT in uppercase",
    });
  },
  "Assert the response contains only capital letters"
);
```

### 2. Estructura del Contexto de Ejecución (`TestContext`)
Conserva el estado a lo largo del escenario. Si necesitas extender variables temporales, utiliza el mapa `ctx.variables` o almacena datos en `ctx.history` para mantener conversaciones multi-turno.
